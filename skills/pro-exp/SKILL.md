# Pro Exp

在以下情况触发：
- 用户希望把最近的 Bug、架构讨论、踩坑经验整理为可复用经验。
- 用户要求在项目内生成 `.exp/` 经验文档。
- 用户说 "记个经验 / 沉淀一下 / 写成文档留给以后 / Human+Agent 双份"。

## 协同与互斥

- 与 `pro-rule` 区分：本 skill 记录 **通用解决方案经验**；`pro-rule` 记录 **用户偏好 / 工作区硬性指令**。
- 链接检索委托 `use-internet`；不要在本 skill 里重复搜索协议。

执行要求：
1. 根据当前上下文提炼一个具体、收敛的主题词，不要使用过长或过泛的文件名。
2. 确保项目根目录存在 `.exp/` 目录，不存在则创建。
3. **先列出 `.exp/` 已有文档**：若存在同主题（或明显同义）文档，优先合并更新内容、追加新章节，不重复新建同名变体；不要使用 `_v2` / `_new` 之类后缀。
4. 一次性生成两份独立 Markdown 文件：
   - `.exp/Human_[主题词].md`
   - `.exp/Agent_[主题词].md`
5. Human 文档要求：
   - 使用大白话、第一人称视角。
   - 像资深工程师带徒弟一样解释。
   - 主目标是沉淀“完整解决方案经验”，而不是只罗列踩坑记录。
   - 内容排序必须遵循“方案优先，坑点其次”。
   - 必须包含以下结构：
     1. 问题场景与目标
     2. 最终解决方案摘要
     3. 实施步骤与关键改动
     4. 为什么这样设计
     5. 验证方式与预期结果
     6. 常见坑点与错误路径
     7. 知识拓展
     8. 参考资料与链接
6. Agent 文档要求：
   - 使用 Directive 风格。
   - 使用强约束的 `Must` / `Never` 句型。
   - 去掉感叹词与多余心理活动。
   - 主目标是让未来 AI 能直接复用“已验证的完整方案”。
   - 内容排序必须遵循“可执行方案优先，反例与踩坑其次”。
   - 文件开头必须包含：
     `> **[系统指令：最高优先级 (AI DIRECTIVE)]**`
   - 必须包含：
     - Context 极简概要
     - Solution Blueprint
     - Execution Steps
     - Verification
     - Rules & Anti-Patterns
     - `Never`
     - `Best Practice`
     - Reusable References
   - 若给出代码片段、命令或配置模板，需要解释其容错或设计理由。
7. 文档整体写作原则：
   - 先回答“正确做法是什么”，再补充“不要怎么做”。
   - 先给可落地方案，再给背景原理。
   - 让读者读完后可以直接照着做，而不是只知道哪里会出错。
   - 如果一个主题已经有稳定解法，坑点部分控制在辅助篇幅，不要喧宾夺主。
8. 两份文档都必须追加“参考资料与链接”部分，用来沉淀后续高频可复用资料，避免未来 AI 反复重复搜索。
9. 参考资料与链接的整理规则：
   - 优先级顺序：
     1. 官方文档
     2. 官方仓库中的 issue / PR / discussion
     3. 权威博客或维护者说明
     4. 高质量社区资料
   - 每条链接至少包含：
     - 标题
     - URL
     - 为什么相关
     - 适用场景
     - 一句话关键结论
   - 若是 Agent 文档，必须额外标注“下次遇到什么问题优先看这条”。
10. 如果当前上下文里已经出现过有价值的文档、仓库页面、Issue、PR 或教程链接，优先直接引用这些现成来源。
11. 如果当前上下文没有足够链接，但要形成长期经验文档，应进行一次有边界的补充检索，把最值得保留的链接写入文档；避免把搜索过程本身重复留给未来会话。
12. 链接部分要追求“少而精”，默认 3 到 8 条，避免堆砌大量低质量结果。
13. 生成完成后，仅汇报两个文件的准确路径与阅读建议，不再主动展开额外探索。
