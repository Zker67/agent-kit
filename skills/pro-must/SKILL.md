# Pro Must

在以下情况触发：
- 用户要求必须按其指定方法执行。
- 用户明确拒绝替代方案或绕路做法。
- 用户说 "按我说的做 / 不要换方案 / 不要自作主张 / 就这么干"。

## 协同与互斥

- 与 `pro-idea` **互斥**：启用期间不主动给替代/扩展建议。
- 与 `pro-explain` 冲突时优先（不得小幅重构）。
- 与 `pro-copy` / `pro-struct` 并用时，其提案只作备选，未经用户同意不采纳。

## 执行要求

1. 执行前**先用一句话复述**用户原始路线，确认理解一致，再开始实现。
2. 以用户原始方案为第一优先级，不要擅自换路线。
3. 即使实现更难，也先按用户要求推进。
4. 只有在客观上完全不可行、风险不可接受或环境不支持时，才说明无法照做。
5. 若必须偏离用户方案，必须清楚说明阻断点、证据和可选退路。
6. 不要把"我觉得更好"当成偏离用户方案的理由。

## 输出要求

- 开头给 `我理解的方案 = ...` 确认句，必要时请用户修正。
- 执行中只报告进度和已做改动，不插入 "我建议换成 X" 类话术。
- 遇到阻断时按四段回复：
  - **阻断点**：具体技术/业务/环境限制。
  - **证据**：报错、日志、官方文档说明。
  - **可选退路**：保持用户意图的最小变形方案。
  - **需要你决定**：明确选项让用户拍板。
