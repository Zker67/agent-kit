# Pro Struct

在以下情况触发：
- 用户希望重构目录结构。
- 用户希望提升组件化、复用性和可维护性。
- 用户说 "目录太乱 / 整理一下结构 / 抽个组件 / 这文件没人用了吧"。

## 协同与互斥

- 搬动前后建议跑 `pro-test` 控制 regression。
- 与 `pro-must` 并用时先给提案再执行，用户同意后再动手。
- 与 `pro-explain` 解耦：本 skill 关注结构；解释代码意图交给 `pro-explain`。

## 执行要求

1. 先分析当前目录结构、模块边界与依赖关系。
2. 提出分步骤整理方案，再逐步执行。
3. 每一步都尽量保持行为不变，控制 regression（回归）。
4. 对疑似废弃文件进行验证（grep 引用、跑测试、看 CI），不要凭猜测移动。
5. 确认确实不再使用后，按项目既有约定存放（`unuse/` / `archive/` / `deprecated/` 等）；没有既有约定时默认用 `unuse/`。
6. 结构调整后要同步修正引用、导入路径、文档与 CI 配置。

## 输出要求

以 "重构提案 + 执行日志" 两段提交：

### 一、重构提案

- **当前结构**：关键层级树 + 问题清单。
- **目标结构**：新目录树 + 分层依据。
- **迁移步骤**：每步的搬运清单、import 修正点、测试验证点。
- **回滚方法**：失败时如何回到当前状态（git 提交点 / 备份方式）。

### 二、执行日志（实际搬动时）

- 每步执行后列出 **已搬文件 / 修正的 import / 跑过的测试 / 结果**。
- 遇阻断立即停下，按 `pro-must` 四段格式回报。
