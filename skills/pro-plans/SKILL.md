# Pro Plans

在以下情况触发：

- 用户要求“写计划到 `plans/` 下”、“先写计划”、“做迁移计划”、“拆分计划”、“实施方案”、“runbook”。
- 用户明确说“只写计划 / 不改代码 / 暂不展开 / 先规划 / 计划完成后再执行”。
- 用户要求新增、移动、归档计划，或同步 `plans/README.md`。
- 用户要求把想法、方案、风险、验证、回滚或阶段拆分沉淀为项目计划文档。

## 协同与互斥

- **与 `pro-summary` 区分**：`pro-plans` 负责写作和维护 `plans/`；`pro-summary` 只负责文档一致性审查时检查 `plans/README.md` 是否与实际计划一致。
- **与 `pro-struct` 协同**：结构重构计划可由 `pro-plans` 先落计划，再由 `pro-struct` 按计划执行迁移。
- **与 `pro-memory` 区分**：计划是项目内可追踪的执行方案；长期上下文和会话归档交给 `pro-memory`。
- **默认不执行计划**：除非用户明确切到执行阶段，否则只写计划和索引，不改业务代码。

## 一、目录约定

1. 所有计划默认写在当前项目根目录的 `plans/` 下。
2. `plans/README.md` 是唯一计划索引；新增、移动、归档计划时必须同步更新。
3. 没有既有约定时，非平凡计划默认使用目录形式：

```text
plans/
├── README.md
└── YYYY-MM-DD-short-topic/
    ├── 00-overview.md
    ├── 01-phase-name.md
    ├── 02-phase-name.md
    └── 09-implementation-report.md
```

4. 简单记录可用单文件 `plans/YYYY-MM-DD-short-topic.md`，但如果计划需要阶段、验证或回滚，优先建目录。
5. 若仓库已有不同命名规范，优先沿用仓库现状，不强行改名。

## 二、计划内容

`00-overview.md` 通常包含：

- 背景与目标。
- 范围和非目标。
- 当前事实与不变量。
- 假设与风险。
- 阶段索引。
- 验证标准。
- 回滚方案。
- 当前状态。

阶段文档通常包含：

- 目标。
- 涉及文件或模块。
- 具体步骤。
- 成功标准。
- 失败分支和回滚点。
- 待确认问题。

实施报告通常包含：

- 实际改动。
- 验证结果。
- 偏离原计划的地方。
- 后续遗留项。

## 三、状态标注

默认使用简单、可 grep 的状态：

| 标注 | 含义 |
|---|---|
| `[planned]` | 已规划，未执行 |
| `[in-progress]` | 正在执行 |
| `[partial]` | 核心可用但有遗留问题 |
| `[done]` | 已完成并验证 |
| `[dropped]` | 已废弃或被替代 |

若项目已使用 `✅ / ⚠️ / 📋 / ❌` 等状态标注，继续沿用现有格式。

## 四、执行边界

1. 用户说“只写计划 / 不改代码 / 暂不展开”时，严格停在计划文档和索引更新。
2. 高风险任务（生产迁移、数据迁移、权限、安全、schema、跨模块重构）先写计划，再自审计划，等待用户确认后再执行。
3. 用户后续明确要求执行时，把计划当作执行合同；若发现计划与当前事实冲突，先回写计划或向用户说明偏差。
4. 执行完成后如用户要求收口，更新计划状态和实施报告，但 commit、push、发布不属于本 skill 的默认动作。

## 五、索引维护

更新 `plans/README.md` 时：

- 保持它作为计划唯一入口。
- 每个计划至少有：相对链接、状态、简短说明。
- 链接必须指向真实存在的文件，通常是 `00-overview.md`。
- 计划状态变化后同步更新索引，避免正文已完成但索引仍是待执行。
- 废弃计划不要删除历史，标记为 `[dropped]` 并说明替代计划。

## 六、输出要求

最终回复包含：

- 新增或更新的计划文件路径。
- `plans/README.md` 是否已同步。
- 当前阶段是否仅计划、已拆分、可执行或已执行。
- 需要用户确认的风险、假设或下一步。
