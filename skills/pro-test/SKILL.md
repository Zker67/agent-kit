# Pro Test

在以下情况触发：
- 用户要求运行测试、调试验证或协助完成测试流程。
- 用户希望在能自动执行时由代理直接完成测试。
- 用户说 "跑个测试 / 看看有没有坏的 / 修一下 / 验证下能不能通"。

## 协同与互斥

- 与全局 `Bug fixing discipline` 对齐：root cause 优先、最小改动、加回归、**不删 / 不弱化已有测试**。
- 大规模重构前后与 `pro-struct` 配合使用。
- 与 `pro-must` 并用时：修复方案必须落在用户指定路线上，不得扩散式改动。

## 执行要求

1. 先识别项目可用的测试、构建、运行与调试入口（`package.json` scripts / `pytest` / `go test` / Makefile 等）。
2. 制定完整测试流程，包括自动测试与必要的手动验证。
3. 可以自行执行的部分，直接执行并记录结果；需要授权才能跑的命令先列出让用户确认。
4. 需要用户参与的部分，明确区分 "我已经做了什么" 和 "你需要做什么"。
5. 若发现 Bug：
   - **先定位 root cause，再最小化修复**；不做无关重构。
   - **添加 / 更新回归测试**覆盖失败场景；在修复点附近补断言。
   - **不得删除或弱化已有测试**来让用例变绿；若测试本身错误，必须单独说明理由。
   - 修复后**重新跑相关测试**直到通过。
6. 输出可复制的验证命令，并说明工作目录与前置条件。

## 输出要求

按四段结构回复：

- **我已执行**：跑过哪些命令 / 测试，结果是什么（通过 / 失败 / 跳过），耗时与关键输出。
- **需要你执行**：依赖你确认的验证步骤，带 `cwd` 与预期结果。
- **可复制命令**：完整 shell 命令块，明确 `cwd` / 环境变量 / 依赖前置。
- **遗留问题**（若有）：阻断点、怀疑方向、下一步建议。
