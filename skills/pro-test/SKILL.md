# Pro Test

在以下情况触发：
- 用户要求代理自主运行测试、调试、验证并修复失败。
- 用户希望代理挂机处理测试，不在每轮失败后停下来等待确认。
- 用户说 "跑个测试 / 挂机跑测试 / 测到通过 / 失败就修 / 修一下 / 验证下能不能通"。

## 协同与互斥

- 与全局 `Bug fixing discipline` 对齐：root cause 优先、最小改动、加回归、**不删 / 不弱化已有测试**。
- 大规模重构前后与 `pro-struct` 配合使用。
- 与 `pro-must` 并用时：修复方案必须落在用户指定路线上，不得扩散式改动。

## 执行要求

1. 先识别项目可用的测试、构建、运行与调试入口（`package.json` scripts / `pytest` / `go test` / Makefile 等），并定义本轮通过标准。
2. 从最小、最快、最相关的测试开始执行，再逐步扩大到类型检查、lint、受影响包构建和必要 smoke test。
3. 对范围内的非破坏性命令和代码修复直接执行，不因一次测试失败停下来，也不为每轮重跑重复请求确认。
4. 若发现 Bug：
   - **先定位 root cause，再最小化修复**；不做无关重构。
   - **添加 / 更新回归测试**覆盖失败场景；在修复点附近补断言。
   - **不得删除或弱化已有测试**来让用例变绿；若测试本身错误，必须单独说明理由。
   - 修复后先重跑失败测试，再重跑受影响测试链路。
5. 持续执行“测试 → 定位 → 修复 → 回归”循环，直到达到通过标准，或遇到缺少凭据、外部服务不可用、需要破坏性操作、必须人工交互等真实阻断。
6. 遇到真实阻断时，先完成所有仍可独立推进的检查，再一次性报告阻断证据和所需输入。
7. 输出最终验证命令、工作目录、关键结果和仍未覆盖的验证范围。

## 输出要求

完成闭环后按三段结构回复，不逐轮堆叠过程日志：

- **结果**：最终是否全绿；修复了什么根因。
- **验证**：执行过的关键命令、通过项、失败项和未运行项。
- **阻断或遗留**（若有）：具体证据、已排除方向和继续推进所需条件。
